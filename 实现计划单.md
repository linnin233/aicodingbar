# ClaudeMonitor 实现计划单

## 核心原则

**聚焦任务栏，有限区域内展示最多内容。** 任务栏是信息瓶颈——宽约 200-400px，高约 40px。所有设计决策优先服务于"一眼看清所有 session 状态"。

## 2. 核心模块实现顺序

### 第 1 步：状态引擎 + HTTP 服务（骨架）

**目标：** 能接收 hook 事件，解析 JSON，维护 session 列表。

- `HttpStateServer.cs` — `HttpListener` 监听 `127.0.0.1:23333`
  - `POST /state` → 解析 JSON body → `StateEngine.ProcessEvent()`
  - 自动端口冲突检测（23333-23337 依次尝试）
  - 端口写入 `~/.aicoding-bar/runtime.json` 供 hook 发现
- `StateEngine.cs` — 核心状态机
  - `ProcessEvent(json)` → 查 ConfigManager 的映射表 → 更新 session 状态
  - `Sessions: ConcurrentDictionary<string, SessionState>` — session_id 为 key
  - 每个 session 维护：状态、开始时间、最近事件、最近工具名、CWD、session_title
  - `CleanupDeadSessions()` — `ProcessHelper.kill(pid, 0)` 探测，死进程自动清理
  - 事件 `OnStateChanged(sessionId, oldState, newState)` 触发 UI 刷新
- `SessionTracker.cs`
  - 单个 session 的状态机：`idle → thinking → working → attention → idle → ...`
  - 记录状态持续时间 `Stopwatch`

### 第 2 步：Hook 脚本（精简 clawd-hook.js）

**目标：** 把 clawd-hook.js 拆出一个最小版本，只做事件上报。

**精简策略：**
- 删除：transcript 解析、session title 提取、tool fingerprint、WT HWND 检测、editor 检测
- 保留：`EVENT_TO_STATE` 映射、进程树追溯（`shared-process.js` 中的 `createPidResolver`）、`postStateToRunningServer`
- 新增：端口发现逻辑（读 `~/.aicoding-bar/runtime.json`）
- `install.js` — 增量修改 `~/.claude/settings.json`，注入 command hook

**注入的事件（与 clawd-on-desk 一致）：**
```
SessionStart, SessionEnd, UserPromptSubmit,
PreToolUse, PostToolUse, PostToolUseFailure,
Stop, SubagentStart, SubagentStop,
Notification, Elicitation
```

### 第 3 步：任务栏渲染（核心）

**目标：** 在 Windows 任务栏上绘制 Claude 状态信息，信息密度最大化。

#### 3.1 任务栏嵌入机制

使用 Windows Taskbar Toolbar（`SHAppBarMessage` 或直接拿到任务栏 HWND 嵌入子窗口）：

```csharp
// 核心思路：找到 Shell_TrayWnd，在其内创建一个透明子窗口
// 用 WPF Window 或直接用 GDI+ 绘制文字
```

具体做法（参考 TrafficMonitor 的 TaskbarWindow 实现）：
1. `FindWindow("Shell_TrayWnd", null)` → 拿到任务栏窗口句柄
2. `FindWindowEx(taskbarHwnd, IntPtr.Zero, "ReBarWindow32", null)` → 任务栏容器
3. 在任务栏右侧区域创建一个小窗口，设为任务栏的子窗口
4. 用 GDI+ 或 DirectWrite 直接在 DC 上绘制文字

**备选方案：** 使用 `Shell_NotifyIcon` 托盘图标 + Tooltip 文字，但这无法做到 TrafficMonitor 那种"直接在任务栏显示"的效果。优先方案仍是子窗口嵌入。

#### 3.2 信息密度设计（重点）

任务栏可用宽度约 200-400px（取决于图标数量），需要最大化信息密度。**纯文字，无图标无 emoji，** 靠颜色 + 缩写传达信息。

**默认模式：紧凑缩写（CompactPillsMode）**

格式：每个 session = 编号 + 2-3 字母状态缩写，空格分隔

```
1:WORK  2:IDLE  3:ERR  4:THK  5:JUG  6:SWP
```

- `1` `2` `3` — 按 session_start 时间排序的序号（去掉 `S` 前缀省空间）
- 状态缩写（2-4 字母）：
  | 状态 | 缩写 | 颜色 |
  |------|------|------|
  | idle | IDLE | 灰 #888 |
  | thinking | THK | 橙 #E8A000 |
  | working | WORK | 蓝 #0080E0 |
  | error | ERR | 红 #E04040 |
  | attention | ATTN | 金 #D0B000 |
  | juggling | JUG | 紫 #9050C0 |
  | sweeping | SWP | 棕 #A06030 |
  | sleeping | SLP | 暗灰 #555 |
  | notification | NOTI | 粉 #E06090 |
- 每个块 `1:WORK` = 6 字符，9pt Consolas 约 36px，300px 可显示 7-8 个 session
- 颜色直接应用到文字本身（`TextRenderer.DrawText` + SolidBrush），背景透明融入任务栏

**备选：聚合摘要（AggregateMode）**

适合 session > 8 个时自动切换或手动选择：

```
W:2  I:1  E:0  T:1  J:0  S:0  N:0
```

- 按状态归类计数，每一项 = 缩写:数量
- 数量为 0 的状态可配置隐藏，进一步省空间
- 约 30 字符，300px 刚好容纳

**备选：高亮模式（HighlightMode）**

```
WORK S1 +2
```

- 只显示最近活跃的 session 状态 + 其余计数
- 最省空间，适合任务栏空间紧张时

**模式切换：** 任务栏区域右键菜单快速切换，配置文件持久化当前选择。也可设置阈值：session ≤ 7 自动用紧凑模式，> 7 自动用聚合模式。

#### 3.3 DPI 适配

- `GetDpiForWindow(taskbarHwnd)` 获取任务栏实际 DPI
- 字体大小按 DPI 缩放：基准 9pt @ 96 DPI
- 窗口尺寸跟随任务栏高度（小任务栏 vs 大任务栏）

#### 3.4 夜间模式适配

- 检测 Windows 注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Themes\Personalize\AppsUseLightTheme`
- 深色模式：亮色文字 + 半透明深色背景
- 浅色模式：深色文字 + 半透明浅色背景

### 第 4 步：Dashboard 窗口

**目标：** 点击任务栏区域 → 弹出 WPF 窗口，表格展示所有 session 详情。

- `SessionGrid.xaml` — WPF DataGrid
  - 列：Session ID(截断) | 名称 | 状态(带颜色圆点) | 持续时间 | 最近工具 | CWD
  - 可排序（点列标题）
  - 可过滤（下拉：全部/活跃/已结束）
  - 右键菜单：复制 session_id / 打开 CWD
- 窗口定位：任务栏上方居中弹出
- 可固定：toggle 按钮，固定后失去焦点不关闭，作为常驻副屏监控面板
- 自动刷新：`StateEngine.OnStateChanged` → `Dispatcher.Invoke` 更新 DataGrid

### 第 5 步：配置系统

**目标：** JSON 配置文件 + 可视化编辑面板。

- `ConfigManager.cs` — 加载/保存 `~/.aicoding-bar/config.json`
- `DefaultConfig.json` — 内置资源，首次启动时写入
  - 默认映射 = clawd-on-desk 的 `EVENT_TO_STATE` 映射表
  - 每个映射项：事件名 → { 状态名, 缩写(2-4字母), 颜色(hex), 优先级 }
- 配置面板（Dashboard 窗口的另一个 Tab）：
  - 事件→状态映射表格，可编辑
  - 任务栏模式选择下拉框
  - 颜色选择器
  - 重置为默认按钮

### 第 6 步：预研通道（可选）

不影响主流程的前提下实现：

- **Named Pipe：** `NamedPipeServerStream` 替代 HttpListener，hook 脚本用 `net.createConnection` 通信
- **File Watch：** `FileSystemWatcher` 监听 `~/.aicoding-bar/state.json`，hook 写文件，C# 读文件
- **Process Poll：** `ManagementEventWatcher` 或定时器扫描 `claude.exe` / `node.exe --claude-code`

四种通道做成 `IStateReceiver` 接口，配置里切换，互不影响。

## 3. 关键技术点

### 3.1 任务栏子窗口稳定性

Windows 任务栏不是为第三方子窗口设计的，每次 Windows 更新或任务栏重绘可能导致子窗口消失。对策：
- 用 `SetTimer` 每秒检查子窗口是否还在，不在则重新创建
- 监听 `WM_TASKBARCREATED` 广播消息（explorer.exe 重启时）

### 3.2 与 clawd-on-desk 的端口共存

两个应用都想用 23333 端口。对策：
- ClaudeMonitor 优先用 23333，被占用就 23334→23337
- Hook 脚本读两个 runtime.json（`~/.clawd/runtime.json` 和 `~/.aicoding-bar/runtime.json`），分别 POST
- 或者用户选择只用其一

### 3.3 单实例

`Mutex("Global\\ClaudeMonitor")` 确保只运行一个实例，重复启动时弹出已有窗口。

## 4. 不在范围内的（不做）

- 桌面悬浮窗（TrafficMonitor 有，但这里聚焦任务栏）
- 多 agent 支持（先只做 Claude Code，架构保留扩展点）
- 历史记录持久化（session 结束就从内存移除，不做数据库）
- GPU 温度/网速等硬件监控（TrafficMonitor 的领域，不重复）
- 远程机器 Claude 状态监控
